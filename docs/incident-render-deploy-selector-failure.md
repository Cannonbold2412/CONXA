# Incident Diagnosis — `render / create-a-service-from-github` fails at step 1 under Claude MCP

**Date:** 2026-06-21
**Skill:** `render:create-a-service-from-github` (a.k.a. `deploy-a-service-from-github`)
**Symptom:** `Step 1 (click) failed: locator.click: Timeout 700ms exceeded` waiting for
`label:has-text("Projects Search CTRL + K K") + button`, then the runtime surfaces
"Layer 4 — vision recovery" / "Layer 5 — intent recovery" to Claude.
**Status:** Root cause identified. The runtime executor (`run.js`) is **byte-identical** across
the commit you flagged (`36a0dc9..HEAD`) and Playwright is unchanged — so this is **not a logic
regression**. The skill's primary selectors were *always* broken; the runtime's text fallback
(`text="New"`) silently rescued them every run. It broke now because the **live Render page opened
a blocking intro/empty-state modal** over the target, so neither the primary selector nor any
recovery candidate can click it within the 700 ms timeout. See §4 for the full regression trail.

---

## 1. TL;DR

The skill package shipped to the runtime contains **structurally broken primary selectors**.
Step 1's selector is:

```css
label:has-text("Projects Search CTRL + K K") + button
```

This can never match the live Render dashboard, because:

1. **There is no `<label>` element** with that text. Render's toolbar buttons are `<button>`
   elements, not form-label/input pairs. `label:has-text(...) + button` describes a DOM shape
   that does not exist on the page.
2. **The "label" text is garbage** — `"Projects Search CTRL + K K"` is *concatenated text
   scraped from several sibling/ancestor nodes* (the "Projects" nav heading + the
   "Search ⌘K" button), not the accessible name of the target ("New"). Step 8's selector is
   even worse: `label:has-text("M My Workspace New Blueprint Search CTRL + K K New Upgrade K") + button`
   — that is the entire top navigation bar's text content.

Because the primary selector matches nothing, every step that uses this pattern times out, and
the runtime's recovery cascade cannot rescue it for this package (see §3). The result is a hard
failure that the MCP server escalates to Claude as "Layer 4/5 recovery", which is why you saw
Claude being asked to fix the selector by hand.

## 2. How execution actually works (so the diagnosis is grounded)

There is exactly **one executor**: `conxa-runtime.exe`, the MCP stdio server built from
`runtime/server.js` + `runtime/run.js`.

- When you run the skill from **Claude Desktop**, Claude calls the MCP tool `execute_skill`.
- When you "Test Plugin" in the **Build Studio**, `conxa-builder/python/conxa_compile/conxa_runtime.py`
  *spawns the same `conxa-runtime.exe`* and calls the same tool over JSON-RPC
  (`call_runtime_tool`, `conxa_runtime.py:240`). It is a client, **not** a second engine.

So "it ran with runtime.exe but fails on MCP" is **the same binary running the same package
both times.** There is no second, richer execution path that the MCP route bypasses.

### What the runtime loads

`runtime/server.js:882-895`:

```js
const rawExec  = JSON.parse(fs.readFileSync("execution.json"));   // bare {type, selector, value}
const rawRec   = JSON.parse(fs.readFileSync("recovery.json"));    // {steps:[{selector_context, anchors,...}]}
const rawSteps = Array.isArray(rawExec) ? rawExec : rawExec.steps;
const steps    = enrichStepsWithRecovery(rawSteps, rawRec);       // merges recovery into each step
```

`execution.json` for this skill is the bare format — **only `type` and `selector`** per step
(generated by `plugin_builder._saved_step_to_execution_step`, `plugin_builder.py:293-340`):

```json
[{ "type": "click", "selector": "label:has-text(\"Projects Search CTRL + K K\") + button" }, ...]
```

`recovery.json` carries the alternatives and text anchors
(`plugin_builder.py:716-718`):

```json
"selector_context": {
  "primary":      "label:has-text(\"Projects Search CTRL + K K\") + button",
  "alternatives": ["... ~ button", "text=\"New\"", "input[type=\"button\"]", "[role=\"button\"][name=\"New\"]"]
}
```

### The recovery cascade (`runtime/run.js`)

For each step `runPlan` (`run.js:705`) does:

1. Run the primary selector with **`ACTION_TIMEOUT_MS = 700`** (`run.js:17`). On failure →
2. `recoverStep` (`run.js:671`), in order:
   - **Tier 1 — compiled alternates:** `compiledSelectors(step)` = `step.compiled_selectors[1:]`
     (`run.js:252`).
   - **Tier 2 — a11y role/name:** `recoverWithA11y` reads `step.element_fingerprint`
     (`run.js:584`) and builds `role=…[name="…"]`.
   - **Tier 2.5 — transient retry** of the primary after 250 ms.
   - **Tier 3 — fallback selectors:** the recovery.json `alternatives` (merged into
     `step.candidates`) + text variants + anchors (`run.js:607`, `fallbackSelectors` `run.js:276`).
   - **Tier 3b — dialog-scoped** primary, **Tier 3c — fuzzy text** match.
3. If nothing recovers → `_buildFailureResponse` (`server.js:677`) emits the screenshots +
   interactive-element dump labelled "Layer 4 / Layer 5" and hands it to Claude.

## 3. Root cause analysis

### 3.1 Primary cause — the compiler emits and *prefers* a brittle `label:has-text()` selector

`generate_stable_selector` (`conxa-builder/python/conxa_compile/compiler/v3.py:361`) adds, for any
element whose enrichment produced a non-empty `label_text` (`v3.py:386-389`):

```python
rows.append(("label", f'label:has-text("{escaped_label}") + {tag_hint}'))
rows.append(("label", f'label:has-text("{escaped_label}") ~ {tag_hint}'))
```

These rows are then ranked by `rank_labeled_selector_candidates`
(`selector_score.py:69`). The scorer (`score_selector_row`, `selector_score.py:21`) gives every
kind a flat base score and breaks ties by `_KIND_PRIORITY` (`selector_score.py:10`):

```python
_KIND_PRIORITY = { "aria": 0, "label": 1, "name": 2, "text_based": 3, "css": 4, "role": 5, "xpath": 6 }
```

So `label` sits second-highest, above `name`, `text_based`, `css`, `id`. **The scorer has no
guard for the two things that make a `label:has-text()` selector worthless:**

- It never checks that a `<label>` element actually exists / is the correct relationship.
- It never penalises a **long, multi-word `label_text`** — the unmistakable signal that the
  enrichment scraped concatenated ancestor text instead of a real accessible name.

Result: garbage like `label:has-text("Projects Search CTRL + K K") + button` is chosen as the
**primary** selector, beating the perfectly good `text="New"` and `input[name="name"]` that exist
in the same candidate set.

### 3.2 Upstream cause — `label_text` is mis-derived as concatenated text

The compiler is downstream of a recorder/enrichment step that computed
`label_text = "Projects Search CTRL + K K"` (and the whole nav bar for step 8). A correct
accessible-name computation for that button would be `"New"`. The label-association /
accessible-name logic grabbed sibling+ancestor text. This is the true origin of the bad data;
the selector generator simply trusted it.

### 3.3 Contributing cause — the data-only package neuters the strongest recovery tiers

The full compiler path *does* produce multi-signal identity — `element_fingerprint` and
`compiled_selectors` exist in `compiler/build.py`, `editor/workflow_service.py`,
`packages/conxa-core/conxa_core/models/skill_spec.py`, and `compiler/patch.py`.

**But the saved-skill export that built this package does not carry them into `execution.json`.**
`_saved_step_to_execution_step` (`plugin_builder.py:293-340`) emits only `{type, selector, value, url}`,
and `enrichStepsWithRecovery` (`run.js:292-321`) only backfills `candidates`, `fallback_selectors`,
`anchors`, `_intent`, `_visual_ref` from recovery.json — it **never sets `element_fingerprint`
or `compiled_selectors`.**

Consequence for this package:

- **Tier 1 (compiled alternates)** — `step.compiled_selectors` is undefined → tier is a no-op.
- **Tier 2 (a11y role/name)** — `step.element_fingerprint` is undefined → `role`/`name` are
  empty → tier produces zero attempts (`run.js:585-588`).

So the two highest-quality recovery tiers are **dead for this package**, leaving only weak CSS
alternatives, text= anchors, dialog-scope, and fuzzy text. For step 1 those alternatives
(`text="New"`, `[role="button"][name="New"]`) either don't match the live DOM cleanly or are
blocked (see 3.5), so recovery exhausts and the step fails hard.

### 3.4 Contributing cause — 700 ms action timeout

`ACTION_TIMEOUT_MS = 700` (`run.js:17`) is aggressive. The primary attempt burns 700 ms on a
selector that will never match, and each fallback attempt is similarly short. On a slow render or
an element that is present-but-not-yet-actionable, this converts "would-have-worked-in-1.5 s" into
a timeout. It compounds every other weakness above.

### 3.5 Contributing cause — live page state diverged from the recording

The "Layer 5" interactive-element dump shows the dashboard had a **modal/intro overlay open**:
`"Close this dialog"`, `"Create your first project"`, `"Close project intro"`, `"Close modal"`.
The recording was captured in a different state (no overlay), and the workflow has **no step to
dismiss it**. An overlay can intercept pointer events on the real target, so even a *correct*
selector for "New" can time out as "not actionable". This is a classic record-time vs run-time
state mismatch the package does not handle.

## 4. "It ran the same `execution.json` perfectly before `36a0dc9` — what changed?"

This is the regression question, and the honest answer surprised me, so here is the evidence
trail rather than a guess.

### 4.1 The executor that resolves selectors did **not** change across `36a0dc9..HEAD`

- `git diff 36a0dc9 HEAD -- runtime/run.js` is **empty**. `run.js` — the step executor, the
  700 ms timeout, and the *entire* recovery cascade — is **byte-identical** before and after the
  commit you named.
- `playwright` is pinned to **`1.59.0`** on both sides (`runtime/package.json`). The selector
  engine that interprets `label:has-text(...)`, `text="New"`, etc. is the same.
- The `execute_skill` handler's core — browser launch (`getCachedBrowser`), `runPlan(...)`, and
  the recovery invocation — is unchanged in the `server.js` diff.

The installed pack (`~/.conxa/skill-packs/render/create-a-service-from-github/execution.json`)
contains the **exact same garbage selectors**, and `manifest.json` pins a checksum for it. So:
same bytes in, same executor, same Playwright. **The deterministic part of execution is identical
across the commit.** A pure-logic regression in the runtime is ruled out by `git diff`.

### 4.2 What *did* change in `36a0dc9..HEAD` (and why none of it changes selector resolution)

| Change | Commit cluster | Effect on a `click` step |
|---|---|---|
| Sync gate: `Promise.race([startupSync, 2000ms])` → `if (!syncState.complete) await startupSync` | `4f73420` | First run now **blocks** on sync instead of a 2 s cap. Can re-pull the published pack; does **not** change how a selector resolves. |
| `sync.js` refactor (per-company, 5-min recency skip, parallel download, shorter timeouts) | `4f73420` | Overwrites local files **only on a cloud delta**. Here `pack.json` has **no `last_synced`** → the pack was **never overwritten**. |
| Host/app split + `bytenode` + `bootstrap.js` fallback (`server.jsc`→`server.js`) | `d1ff63c`, `6acb022`, `d002078` | Changes *delivery* (bytecode vs source) and which file the exe boots. `run.js` source is identical, so the running logic is identical. No app-layer is installed, so the bundled code runs. |

None of these touch selector generation or the recovery tiers.

### 4.3 The runtime log shows what actually happened — and it's not a selector problem

`~/.conxa/logs/recovery.log` for the failing runs contains **only**:

```json
{"event":"terminal_failure","slug":"create-a-service-from-github",
 "error":"Step 1 (click) failed: locator.click: Timeout 700ms exceeded ... label:has-text(\"Projects Search CTRL + K K\") + button"}
```

Every recovery tier in `run.js` logs an event **on success** (`tier1_compiled_alt`,
`tier2_a11y`, `transient_recovered`, `layer_recovered`, …). There are **none** — so
`recoverStep` ran through *all* tiers, including the `text="New"` / `[role="button"][name="New"]`
fallbacks from `recovery.json`, and **every one failed**, then it threw `terminal_failure`.

Here's the key inference: the "Layer 5" dump proves the live page **does have a "New" button**
(`{"tag":"button","text":"New","id":"menu-_r_4_"}`). If a correct candidate like `text="New"`
*also* times out, the target is not mis-selected — it is **non-actionable**. The same dump shows
a blocking overlay open: `"Close this dialog"`, `"Create your first project"`,
`"Close project intro"`, `"Close modal"`. A modal/intro overlay intercepts pointer events on the
button behind it, so *neither the primary selector nor any recovery candidate* can click it
inside 700 ms.

### 4.4 Conclusion — what "worked perfectly before" really means

The primary selector `label:has-text("Projects Search CTRL + K K") + button` **never worked** —
it has been garbage since the day the pack was built. When the skill "ran perfectly" before, the
runtime's **Tier-3 fallback was silently rescuing step 1** by clicking `text="New"`. You never saw
the bad primary because recovery absorbed it every time. The skill was therefore *always* fragile:
one signal — a clean, unobstructed "New" button — away from working.

What broke it is **not the commit's code** (the executor is byte-identical). It is the **live
Render page state**: the dashboard now opens an intro / empty-state modal over the toolbar, and
the workflow has no step to dismiss it. With the real target covered, the recovery net that used
to carry the skill can no longer land a click, and the brittle primary has nothing to fall back
onto. The result is the hard failure you see.

> Two honest caveats. (1) The correlation with `36a0dc9` is real but, on this machine, *coincidental
> to the code* — the change there is sync/bundling, not execution; the page-state change is what
> flipped a latently-broken skill into a visibly-broken one. (2) The one scenario in which the
> commit *would* be causally involved: if before `36a0dc9` you were running an **earlier pack
> build with cleaner selectors**, and the new **hard sync gate** then pulled the
> garbage-selector published pack over it. That is plausible in principle, but the evidence here
> argues against it — `pack.json` has **no `last_synced`**, so sync never wrote this pack; it is
> the pack as installed. Either way the fix is the same (below): the skill must not depend on a
> brittle primary + a single text fallback, and it must dismiss blocking overlays.

## 5. Fixes

### 5.1 Immediate — unblock this skill now

Edit `…/render-plugin/skills/deploy-a-service-from-github/execution.json` and replace every
`label:has-text("…") + …` primary with a real selector, then re-run with `resume_from: 0`.
Suggested replacements (cross-checked against the live element dump and recovery.json
alternatives):

| Step | Intent | Replace with |
|---|---|---|
| 1 | click "New" | `button:has-text("New")` (the `text="New"` alt also works) |
| 2 | "Blueprint" link | `a:has-text("Blueprint")` / `text="Blueprint"` |
| 3,4 | repo search input | `[data-testid="select-git-repo-url-input"]` or `input[placeholder*="Search"]` |
| 5 | "Connect" on repo row | `button:has-text("Connect")` scoped to the repo row |
| 6,7 | name field | `input[name="name"]` (already the primary here — good) |
| 8 | "Deploy Blueprint" | `button:has-text("Deploy Blueprint")` |

Also add a **dismiss-overlay step** at the start if the intro modal is present, e.g. a tolerant
click on `button:has-text("Close")` / `[aria-label="Close"]` guarded so it no-ops when absent.

> Note: editing `execution.json` by hand is a stop-gap. The same garbage will return on the next
> rebuild unless the compiler/ranking is fixed (§5.2).

### 5.2 Short-term — stop generating and preferring bad selectors

1. **Penalise/disqualify bad `label:has-text()` in the scorer** (`selector_score.py:21`):
   - Heavily penalise (or drop) `label` rows whose `label_text` is long or multi-word
     (e.g. `> 3` words or `> 40` chars) — a strong signal of scraped ancestor text.
   - Demote `label:has-text()` below `name`, `text_based`, `data-testid`, and `id` in
     `_KIND_PRIORITY`. A synthetic `label + sibling` guess should never outrank a real
     `[data-testid=…]` or `input[name=…]`.
2. **Prefer `data-testid` / stable `id` / role+name.** Render exposes `data-testid`
   (e.g. `select-git-repo-url-input`); these should be first-class high-score candidates in
   `generate_stable_selector` (`v3.py:361`), not buried.
3. **Fix the accessible-name enrichment** so `label_text` for a button is its own accessible name
   ("New"), not concatenated sibling/ancestor text. This is the upstream root cause; fixing the
   scorer is a safety net, fixing enrichment removes the bad input.
4. **Carry multi-signal identity into the data-only package.** Add `element_fingerprint`
   (role, accessible name, tag, key attrs) and `compiled_selectors` to
   `_saved_step_to_execution_step` (`plugin_builder.py:293`) so the runtime's Tier 1/Tier 2
   recovery actually has data to work with. Today those tiers are dead weight for saved skills.
5. **Raise / make adaptive the action timeout.** `ACTION_TIMEOUT_MS = 700` (`run.js:17`) is too
   tight for first-paint-heavy SPAs. Consider 1500–2500 ms for the primary attempt, or a
   wait-for-actionable pre-check before counting a timeout as failure.
6. **Add a generic overlay/dialog-dismiss pre-pass** in the runtime before each interactive step
   (close cookie banners, intro modals) so record-time/run-time state drift doesn't hard-fail.

### 5.3 Long-term — the right architecture

The core architectural flaw is that **element identity is collapsed into a single brittle CSS
string at compile time, and the runtime resolves it as a string.** Self-healing was designed
around multi-signal identity (`element_fingerprint`), but the shipped package throws that signal
away, so recovery degrades to "try a list of CSS guesses."

Target design:

1. **One rich element-identity object per step, carried end-to-end** (recorder → compiler →
   package → runtime), never flattened to a lone CSS string. Minimum signals: role, accessible
   name, tag, text, stable attributes (`data-testid`, `name`, `id` with dynamic-id detection),
   structural anchors, bounding box, and a visual crop. `skill_spec.py` already models
   `element_fingerprint` — **make the package format and the runtime both consume it.**
2. **Resolution is scoring, not first-match.** The runtime should gather *all* plausible
   candidates on the live page and pick the highest-scoring match against the fingerprint
   (name + role + attributes + position + visual), rather than firing CSS strings in priority
   order and taking the first that doesn't throw. This makes "the label selector is wrong" a
   non-event: the fingerprint still resolves the button.
3. **Semantic-first, CSS-as-hint.** Prefer role+accessible-name (`getByRole`) and `data-testid`
   as the *primary* resolution path; treat compiled CSS as one weak signal among many. This
   directly prevents a synthetic `label:has-text()` guess from ever being load-bearing.
4. **State-aware preconditions.** Each step declares the expected precondition (URL pattern,
   required visible element, "no blocking overlay"). The runtime reconciles state (dismiss
   overlays, wait for the right view) before acting — instead of assuming the recorded DOM.
5. **Keep the LLM tiers (4/5) as a true last resort**, fed by the same fingerprint, and **write
   successful LLM resolutions back** as learned candidates so a skill self-heals permanently
   instead of re-paying the LLM cost every run. (Cost note: Tier 1/2 must stay zero-token per the
   `Key Invariants` in `CLAUDE.md`.)
6. **Compile-time selector validation.** Before publishing, replay each selector against the
   captured DOM snapshot and **reject any primary that doesn't uniquely resolve** — a build gate
   that would have caught `label:has-text("Projects Search CTRL + K K") + button` immediately.
